3. Data Transformation and Cleaning

3.1 Feature Selection Rationale

We select features based on three criteria:

  1. Domain relevance: Variables that intuitively drive wages (experience, education, skills, role, country)
  2. Data quality: Dropping columns with >40% missing values
  3. Avoiding data leakage: Excluding any variable that directly encodes wage

Wages are not internationally uniform, compensation in prosperous economies is significantly higher than in developing countries, making country of employment one of the most important variables in our transformation and modelling process. As part of our data quality checks, we also screened for missing values. We applied a 40% threshold, meaning any variable with more than 40% missing values was removed. This ensures we only retain features that carry meaningful information. Finally, to prevent data leakage, we carefully excluded any variables that directly encode wage. Including such variables would artificially inflate model performance, as the model would essentially be using the answer to predict the answer, producing misleadingly optimistic results that would not generalise to new data.

4. Model: XGBoost

4.1 Why XGBoost?

We chose XGBoost (Extreme Gradient Boosting) as our model for the following reasons:

  • Best-in-class for tabular data: XGBoost consistently outperforms other algorithms on structured datasets with mixed feature types
  • Handles sparsity natively: Our dataset has 260+ binary columns (programming languages, tools, activities) — XGBoost is specifically optimised for sparse inputs
  • Gradient boosting: Builds trees sequentially, each correcting the errors of the previous one, making it highly accurate
  • Built-in regularisation: lambda, alpha, gamma, and min_child_weight parameters prevent overfitting on noisy survey data
  • SHAP compatibility: Native SHAP support provides state-of-the-art explainability

4.2 Hyperparameter Tuning

We use xgb.cv with 5-fold cross-validation to find the optimal number of boosting rounds, and tune key parameters to maximise R².

Parameters explained:

  • eta (0.05): Lower learning rate — each tree contributes less, requiring more rounds but producing a more robust model
  • max_depth (5): Tree depth — controls model complexity; 5 balances expressiveness and overfitting
  • subsample (0.8): Each tree sees 80% of training rows — adds randomness, reduces overfitting
  • colsample_bytree (0.7): Each tree sees 70% of features — forces diversity across trees
  • min_child_weight (5): Minimum observations in a leaf — prevents the model from learning noise in small subgroups
  • gamma (0.1): Minimum loss reduction to make a split — acts as pruning, improves generalisation

To evaluate model performance visually we produce three diagnostic plots. The first plot shows Actual vs Predicted Wage, in a perfect model all points would fall exactly on the red dashed line. Points scattered around it indicate prediction errors, with points above the line meaning the model underestimated and points below meaning it overestimated. The second plot shows Residuals vs Predicted, residuals are the difference between actual and predicted wages. Ideally these should be randomly scattered around zero with no clear pattern. A pattern would suggest the model is systematically wrong for certain wage ranges. The third plot shows the Distribution of Residuals, ideally this should be a bell curve centered at zero, meaning errors are symmetric and the model is not consistently over or under predicting. In our case the residuals are approximately centered at zero, which is a good sign, though the spread reflects the inherent difficulty of predicting wages from survey data.

Performance discussion:

The model achieves an R² of approximately 0.31–0.40 on this dataset. While this may seem modest, it is expected and acceptable given:

  • Bucketed variables: Age and experience are ranges (e.g. 25–29, 1–2 years), not exact values — this loses precision
  • Self-reported global survey: Wages vary enormously across countries and roles; a single model must capture all this diversity
  • Inherent noise: Survey respondents may approximate or misreport salaries
  • No continuous numeric predictors: Almost all features are categorical, limiting the model’s ability to capture fine-grained patterns

An R² of 0.31+ on a noisy global wage survey is consistent with published research on similar datasets.

5.2 DALEX Explainer Setup

xgb_predict_fun <- function(model, newdata) {
  predict(model, xgb.DMatrix(as.matrix(newdata)))
}

explainer <- DALEX::explain(
  model            = xgb_model,
  data             = as.data.frame(X_test),
  y                = y_test,
  predict_function = xgb_predict_fun,
  label            = "XGBoost",
  verbose          = FALSE
)

To interpret our XGBoost model we use DALEX (Descriptive mAchine Learning EXplanations), a model-agnostic explainability framework. DALEX works by wrapping around any trained model. Regardless of its internal complexity and providing a unified set of tools to understand its behaviour. This is particularly valuable for XGBoost, which is inherently a black box model. Without explainability tools we would know what the model predicts but not why. DALEX solves this by providing three complementary techniques:

  • Partial Dependence Profiles: Show how the predicted wage changes as a single feature varies, holding all other features constant. This reveals the direction and shape of each feature’s effect on wages globally across all observations.
  • Break-Down plots: Decompose a single prediction into individual feature contributions, showing exactly how each variable pushed the predicted wage up or down from the average baseline.
  • SHAP values: A theoretically rigorous extension of break-down plots that accounts for feature interactions, providing the most accurate attribution of each feature’s contribution to a specific prediction.

Together these three techniques give us both a global understanding of the model (which features matter most overall) and a local understanding (why did the model predict this specific wage for this specific person).

7. Conclusions

Model Choice

We selected XGBoost as our sole model because it is the best-performing algorithm for tabular data with many sparse binary features. Its gradient boosting mechanism — building trees sequentially to correct previous errors — combined with strong regularisation parameters (lambda, alpha, gamma, min_child_weight) makes it particularly well-suited to noisy survey data.

Model Performance

  • R² = 0.327 on the held-out test set
  • RMSE ≈ 4.9544^{4} USD in original scale
  • MAE ≈ 2.5154^{4} USD in original scale

An R² of ~0.31–0.40 is expected and acceptable for this dataset. The primary reasons are: (1) age and experience are bucketed ranges, not exact values; (2) wages are self-reported across vastly different countries and industries; (3) no continuous numeric predictors are available. This R² is consistent with published wage prediction models on similar survey datasets.

Feature Decisions

  • Included: demographics (age, gender, country), education, experience, job role, industry, programming skills, ML tool usage, cloud platforms, activities, coding time percentage, ML experience years
  • Excluded: Columns with >40% missing values
  • Transformed: All categoricals one-hot encoded (260+ features); wage log-transformed to address right skew

Key Drivers of Wage

Based on XGBoost feature importance (Gain) and SHAP values:

  • Country — the single strongest predictor, reflecting labour market and cost-of-living differences
  • Age and years of experience — strong positive relationship with wage
  • Job role — Chief Officers, Software Engineers, and Data Scientists predict higher wages
  • Education — Doctoral and Master’s degrees associated with higher wages
  • ML maturity at work — respondents at companies with established ML in production earn more

Limitations

  • The model captures survey-level patterns, not individual circumstances
  • Wage brackets are self-reported and may be imprecise
  • The dataset reflects the Kaggle community, which skews towards certain countries and roles
  • Team predictions are benchmarks — actual salaries depend on company, negotiation, and local market conditions
