Predictive Modelling for Wages

Objectives

  • Build a model that will predict the variable “wage”. Put differently, the dependent variable of the model should be y = “wage”.

  • You are free to decide on the features you include as explanatory variables as well as the specific models you use. You need to provide details as to decisions you have taken («why did you choose algorithm XYZ»? «How did you choose the features?» «How did you transform them»? Etc.).

  • Once you have built your predictive model, you need to explain it! You need to run explainability techniques so to identify the main features that drive the wage predictions in your model.

  • How will your model predict for you? Use your model to predict the wage that each one from the team will earn in the future.

3. Data Transformation and Cleaning

3.1 Feature Selection Rationale

We select features based on three criteria:

  1. Domain relevance: Variables that intuitively drive wages (experience, education, skills, role)
  2. Data quality: Dropping columns with >40% missing values
  3. Avoiding data leakage: Excluding any variable that directly encodes wage
# Step 1: Remove rows where wage is missing
data_clean <- data %>% filter(!is.na(wage))
cat("Rows after removing missing wage:", nrow(data_clean), "\n")

# Step 2: Drop columns with >40% missing values
missing_threshold <- 0.40
keep_cols <- colnames(data_clean)[colMeans(is.na(data_clean)) <= missing_threshold]
data_clean <- data_clean %>% select(all_of(keep_cols))
cat("Columns remaining after missing filter:", ncol(data_clean), "\n")

4. Model Building

Our primary model is XGBoost, with Random Forest included as a lightweight baseline for comparison.

Why XGBoost as the primary model?

  • State-of-the-art performance on tabular data
  • Handles 260+ sparse binary columns (programming languages, tools, activities) particularly well
  • SHAP explainability integrates natively and produces highly interpretable results
  • Faster and more tunable than Random Forest

Why also run Random Forest?

  • Provides a meaningful benchmark using a different learning mechanism (bagging vs boosting)
  • Demonstrates we evaluated multiple approaches

4.2 XGBoost (Primary Model)

XGBoost requires fully numeric input, so we one-hot encode all factor columns using dummyVars. We then use xgb.cv to find the optimal number of boosting rounds via 5-fold cross-validation.

# One-hot encode factors for XGBoost
dummies   <- dummyVars(log_wage ~ ., data = train_data)
train_xgb <- predict(dummies, newdata = train_data) %>% as.data.frame()
train_xgb$log_wage <- train_data$log_wage

test_xgb  <- predict(dummies, newdata = test_data) %>% as.data.frame()
test_xgb$log_wage  <- test_data$log_wage

# Clean column names for compatibility
names(train_xgb) <- make.names(names(train_xgb), unique = TRUE)
names(test_xgb)  <- make.names(names(test_xgb),  unique = TRUE)

# Prepare XGBoost matrices
X_train <- as.matrix(train_xgb %>% select(-log_wage))
y_train <- train_xgb$log_wage
X_test  <- as.matrix(test_xgb  %>% select(-log_wage))
y_test  <- test_xgb$log_wage

dtrain <- xgb.DMatrix(data = X_train, label = y_train)
dtest  <- xgb.DMatrix(data = X_test,  label = y_test)

# Cross-validation to find best number of rounds
set.seed(42)
xgb_cv <- xgb.cv(
  data    = dtrain,
  nrounds = 200,
  nfold   = 5,
  params  = list(
    objective        = "reg:squarederror",
    eta              = 0.1,
    max_depth        = 6,
    subsample        = 0.8,
    colsample_bytree = 0.8
  ),
  verbose = FALSE
)

best_nrounds <- which.min(xgb_cv$evaluation_log$test_rmse_mean)
cat("Best nrounds:", best_nrounds, "\n")
Best nrounds: 64 
# Train final XGBoost model
set.seed(42)
xgb_model <- xgb.train(
  data    = dtrain,
  nrounds = best_nrounds,
  params  = list(
    objective        = "reg:squarederror",
    eta              = 0.1,
    max_depth        = 6,
    subsample        = 0.8,
    colsample_bytree = 0.8
  ),
  verbose = 0
)

# XGBoost test predictions
xgb_preds_log  <- predict(xgb_model, dtest)
xgb_preds_orig <- expm1(xgb_preds_log)

xgb_rmse_log  <- rmse(y_test,      xgb_preds_log)
xgb_rmse_orig <- rmse(actual_orig,  xgb_preds_orig)
xgb_r2        <- cor(y_test,       xgb_preds_log)^2

cat(sprintf("XGB | RMSE (log): %.4f | R²: %.4f | RMSE (orig): %.1f\n",

5. Model Explainability

We use two complementary techniques:

  • XGBoost Feature Importance: Which features contribute most globally
  • DALEX: Partial dependence profiles and SHAP break-down for individual predictions

DALEX wraps our XGBoost model and provides a unified interface for explainability techniques.

7. Conclusions

Model Performance

  • We trained Random Forest (baseline) and XGBoost (primary model)
  • XGBoost achieved R² = 0.314 vs Random Forest R² = 0.301 on the test set
  • An R² of ~0.31 is expected for this dataset — wages are self-reported, globally diverse, and age/experience are bucketed ranges rather than exact values, all of which add noise
  • We log-transformed wage to handle its right-skewed distribution

Feature Decisions

  • Included: demographics (age, gender, country), education, experience, job role, industry, programming skills, ML tool usage, cloud platforms, activities, coding time
  • Excluded: Columns with >40% missing values
  • Transformed: All categoricals one-hot encoded for XGBoost; wage log-transformed

Key Drivers of Wage

Based on XGBoost feature importance and SHAP values, the top factors driving wages are typically:

  • Country of employment (largest effect — reflects cost of living and market rates)
  • Age bucket and years of experience
  • Job role (e.g. Chief Officer, Software Engineer vs Student)
  • Education level and undergraduate major
  • Python usage and ML maturity at work

Team Predictions

Each team member’s predicted wage reflects their profile’s combination of country, experience, education, and technical skills. These should be interpreted as expected ranges — the model captures population-level patterns from the Kaggle Data Science Survey, not individual-level nuance.
